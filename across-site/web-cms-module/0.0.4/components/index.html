    WebCmsComponentModelSet componentModelSet = new WebCmsComponentModelSet();
    TextWebCmsComponentModel text = new TextWebCmsComponentModel();
    text.setContent("some text");
    text.setName("my-custom-component");
    componentModelSet.add(text);
    componentModelHierarchy.registerComponentsForScope(componentModelSet, "asset");
  }
}</code></pre>

<div class="paragraph">
<p>&lt;div class="sect3"&gt;
&lt;h4 id="components-content-markers"&gt;&lt;a class="anchor" href="#components-content-markers"&gt;&lt;/a&gt;Content markers&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;{module-name} provides content marker infrastructure to work with special marker snippets inside textual content.
Upon rendering, these markers would be processed and the resulting output would be rendered instead.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Example text with content markers&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-text hljs" data-lang="text"&gt;Sample content markers
Insert component content: @@wcm:component(title,global,false)@@
Insert placeholder content: @@wcm:placeholder(myplaceholder)@@&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Two default markers are supported:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;placeholder content markers indicate placeholder content should be rendered at the marker&#8217;s location&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;component content markers indicate a component should be rendered at the marker&#8217;s location&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Content marker format&lt;/div&gt;
&lt;p&gt;A &lt;code&gt;WebCmsContentMarker&lt;/code&gt; consists of 2 parts:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;a key indicating the type of content marker - the key is required and cannot contain whitespace&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;an optional string of parameters that is supported by that type&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In text a content marker is always surrounde by &lt;code&gt;@@&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;div class="title"&gt;Example content markers:&lt;/div&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@my.marker@@&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;key: &lt;em&gt;my.marker&lt;/em&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@my.marker(my, params)@@&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;key: &lt;em&gt;my.marker&lt;/em&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;parameter string: &lt;em&gt;my, params&lt;/em&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The &lt;code&gt;WebCmsContentMarkerService&lt;/code&gt; is the service for working with content markers in general.
Default rendering support is available for Thymeleaf templates, see the &lt;a href="#components-thymeleaf-content-markers"&gt;{module-name} Thymeleaf dialect&lt;/a&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Content markers are supported in all &lt;a href="#components-default-component-types"&gt;default component types&lt;/a&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="sect4"&gt;
&lt;h5 id="components-placeholder-content-markers"&gt;&lt;a class="anchor" href="#components-placeholder-content-markers"&gt;&lt;/a&gt;Placeholder content markers&lt;/h5&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;A placeholder content marker indicates that placeholder content should be rendered at the marker&#8217;s location.
The key is &lt;em&gt;wcm:placeholder&lt;/em&gt; and the parameter is the name of the placeholder whose content should be rendered.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Example: &lt;code&gt;@@wcm:placeholder(some.placeholder)@@&lt;/code&gt;&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect4"&gt;
&lt;h5 id="components-component-content-markers"&gt;&lt;a class="anchor" href="#components-component-content-markers"&gt;&lt;/a&gt;Component content markers&lt;/h5&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;A component content marker indicates a component should be rendered at the location of the marker.
The key is &lt;em&gt;wcm:component&lt;/em&gt; and 3 parameters are required:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;the component name&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;the scope in which to look for the component&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;should parent scopes be searched as well: &lt;strong&gt;true&lt;/strong&gt;/&lt;strong&gt;false&lt;/strong&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Whitespace is not supported in the parameter string.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Examples:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@wcm:component(my.component.name,default,true)@@&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;will search for &lt;em&gt;my.component&lt;/em&gt; in the default scope and all parent scopes&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@wcm:component(my.component.name,global,false)@@&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;will search for &lt;em&gt;my.component&lt;/em&gt; in the global scope only (as global is the last scope the parent flag is of no meaning)&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@wcm:component(my.component.name,container,false)@@&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;will search for &lt;em&gt;my.component&lt;/em&gt; in the container members - only relevant in the context of a &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt; being rendered&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;

&lt;div class="paragraph"&gt;
&lt;p&gt;See &lt;a href="#components-thymeleaf-content-markers"&gt;{module-name} Thymeleaf dialect&lt;/a&gt; on how to add content marker support for component rendering.&lt;/p&gt;

<div class="paragraph">
<p>&lt;div class="sect2"&gt;
&lt;h3 id="components-thymeleaf-integration"&gt;&lt;a class="anchor" href="#components-thymeleaf-integration"&gt;&lt;/a&gt;Thymeleaf integration&lt;/h3&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;WebCmsModule provides its own Thymeleaf dialect for working with components.
It enables you to:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;replace template fragments by component output&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;auto-create components from templates based on the template content&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;define placeholder content&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The Thymeleaf dialect is a very powerful tool to convert your static templates into user modifiable components.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_html_attributes"&gt;&lt;a class="anchor" href="#_html_attributes"&gt;&lt;/a&gt;HTML attributes&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Interacting with components is done by adding attributes to your template elements.&lt;/p&gt;
&lt;/div&gt;
&lt;table class="tableblock frame-all grid-all spread"&gt;
&lt;colgroup&gt;
&lt;col style="width: 50%;"&gt;
&lt;col style="width: 50%;"&gt;
&lt;/colgroup&gt;
&lt;tbody&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:component&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;The body of the element will be replaced with the rendering of the component, if the component is found.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:scope&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Specify the name of the scope in which the component should be looked for - if omitted wil look in the default scope and all parent scopes.
When specified, only that scope will be queried unless &lt;em&gt;search-parent-scopes&lt;/em&gt; is explicitly present.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:search-parent-scopes&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Set explicitly to &lt;code&gt;false&lt;/code&gt; if you only want the component to be looked up in the scope specified (or the default scope) but none of the parent scopes.
Otherwise adding the attribute will ensure that parent scopes are always searched, even if an explicit &lt;em&gt;wcm:scope&lt;/em&gt; was specified.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:always-replace&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;If present the body of the element will always be replaced even if the component is not found.
The actual (optional) value of this attribute is irrelevant.&lt;/p&gt;
&lt;p class="tableblock"&gt;Default behaviour is to render the original template markup if a component is not found, use this attribute to force empty output instead.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:auto-create&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;If present, the component will be auto-created with the element body as input parameters.
 Optionally the attribute value can be be the scope name in which the component should be created.
If no value is present, the value of &lt;em&gt;wcm:scope&lt;/em&gt; will be used to determine the scope in which to create the component.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:type&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Type key of the &lt;code&gt;WebCmsComponentType&lt;/code&gt; of the component.
This will only be used for auto-creation of a component.
If no type is specified, generic markup (&lt;strong&gt;html&lt;/strong&gt;) will be assumed.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:parent-create-include&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Relevant only during auto-creation of a possible parent component.
If the parent component is being auto-created nested components would by default be inserted as content markers or added as &lt;code&gt;ProxyWebCmsComponentModel&lt;/code&gt; members.
By providing this attribute (the actual value is ignored) you indicate the output of the component should be inserted instead.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:parse-placeholders&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;If present on an element attributed with &lt;em&gt;wcm:component&lt;/em&gt;, the body of the element will always be parsed and all sections marked &lt;em&gt;wcm:placeholder&lt;/em&gt; will be made available in the &lt;code&gt;WebCmsPlaceholderContentModel&lt;/code&gt;.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;wcm:placeholder&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Attribute to indicate that a section should be rendered to a placeholder.
 The name of the placeholder is the value of the attribute.
 Note that the element itself will also be part of the placeholder content, not just the body of the element.&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;/tbody&gt;
&lt;/table&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Auto-creation of components from markup&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;if you don&#8217;t specify a component type - the default component type will be created&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;if you don&#8217;t specify a scope on a component - a default scope will be used:&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;outside a parent component the default scope will be &lt;em&gt;asset&lt;/em&gt;, and parents will be searched if not specified within the &lt;em&gt;asset&lt;/em&gt; scope.&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;inside a parent component that is not a container: the default scope will be &lt;em&gt;asset&lt;/em&gt; and parent scopes will be searched if not specified&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;inside a parent component that is a container: the default scope will be the &lt;em&gt;container&lt;/em&gt; and parents will &lt;strong&gt;not&lt;/strong&gt; be searched&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;if your component is a container:&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;any placeholders in the body will be added as placeholder child components and the body markup will refer to those components&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;any scoped component will be added as a proxy component referring to the original scoped component and the body markup will refer to the proxy components&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Special scopes:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;default&lt;/strong&gt;: always refers to the lowest level - first one to perform the lookup (usually asset) - you would not normally use this scope name as it is the default if a scope name is omitted&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;asset&lt;/strong&gt;: components attached to the current asset being rendered, usually an asset specific name (e.g. &lt;strong&gt;article&lt;/strong&gt;) will be available as well, but both will refer to the same components&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;global&lt;/strong&gt;: always refers to the global scope, components that have no owner and are shared across domain&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;domain&lt;/strong&gt;: components that have no owner but are only available on the specific domain&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;container&lt;/strong&gt;: this is a reserved scope name that ensures the component is added as a container member - you would not normally use this scope name as it is the default if omitted inside a container component&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;&lt;strong&gt;NOTE&lt;/strong&gt;
In a single domain configuration, scope &lt;strong&gt;domain&lt;/strong&gt; will always be the same as scope &lt;strong&gt;global&lt;/strong&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_examples"&gt;&lt;a class="anchor" href="#_examples"&gt;&lt;/a&gt;Examples&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;This section contains a number of real-life examples for working with components from Thymeleaf.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;Only look in the scope of the page asset. If not present, don&#8217;t do anything.&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre&gt;&lt;th:block wcm:component="page-header" wcm:scope="page" wcm:search-parent-scopes="false"&gt;
    &lt;div class="l-page<em>title"&gt;Our title&lt;/div&gt;
    &lt;p&gt;
        Our text
    &lt;/p&gt;
&lt;/th&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;Start by looking in the scope of the page asset, don&#8217;t look in the parent components even if you don&#8217;t find them in the page scope. If the component is not found,
render the code that is written down in the thymeleaf template.&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre&gt;&lt;th:block wcm:component="page-header" wcm:scope="page"&gt;
    &lt;div class="l-page</em>title"&gt;Our title&lt;/div&gt;
    &lt;p&gt;
        Our text
    &lt;/p&gt;
&lt;/th&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;Start by looking if the component exists in the page asset, but look in the parent scope(s) if you don&#8217;t find the component in the page scope. If the component
is nowhere to be found, the content in the thymeleaf template itself is rendered.&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre&gt;&lt;th:block wcm:component="page-header" wcm:scope="global"&gt;
    &lt;div class="l-page__title"&gt;Our title&lt;/div&gt;
    &lt;p&gt;
        Our text
    &lt;/p&gt;
&lt;/th&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;Look in the global scope for the component, skipping over any and all more granular scopes. If the component is nowhere to be
found, the content in the thymeleaf template itself is rendered.&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_rendering_a_component"&gt;&lt;a class="anchor" href="#_rendering_a_component"&gt;&lt;/a&gt;Rendering a component&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can specify a component to render by just setting the name of the component as the value of the &lt;em&gt;wcm:component&lt;/em&gt; attribute.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Simple rendering of component&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component"&gt;This markup will only be rendered if that component does not exist.&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In the example above the entire component hierarchy for the request will be searched bottom-up for a component with that name.
Only if that component does not exist will the original template be rendered.
If the component is found, the content of the &lt;em&gt;div&lt;/em&gt; element will be replaced, but the &lt;em&gt;div&lt;/em&gt; element itself will remain.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
You can still use the standard Thymeleaf dialect (eg. &lt;em&gt;th:block&lt;/em&gt; or &lt;em&gt;th:remove&lt;/em&gt;) to manipulate the wrapping element.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Replace the component body if the component is not found&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:always-replace&gt;This markup will never be rendered.&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;By adding the &lt;em&gt;wcm:always-replace&lt;/em&gt; attribute you can ensure the original template markup will be suppressed.
Even if the component is not found, the body of the &lt;em&gt;div&lt;/em&gt; element will be empty.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Specifying a scope for the component&lt;/div&gt;
&lt;p&gt;In our original example all scopes will be traversed bottom-up to find the component.
If you only want to look for the component in a specific scope, simply set the &lt;em&gt;wcm:scope&lt;/em&gt; attribute.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:scope="global"&gt;Replaced by a globally shared component.&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;When set, the component will be looked for only in that scope (unless you also set &lt;em&gt;wcm:search-parent-scopes&lt;/em&gt;).
In our example we look for a component 'my-component' in the set of shared components.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
A &lt;code&gt;WebCmsComponentModel&lt;/code&gt; is also a &lt;code&gt;ViewElement&lt;/code&gt; and can always be rendered using the &lt;strong&gt;across:view&lt;/strong&gt; element from the AcrossWebDialect as well.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect1"&gt;
&lt;h2 id="_auto_create_a_markup_component"&gt;&lt;a class="anchor" href="#_auto_create_a_markup_component"&gt;&lt;/a&gt;Auto-create a markup component&lt;/h2&gt;
&lt;div class="sectionbody"&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The presence of &lt;em&gt;wcm:auto-create&lt;/em&gt; will automatically create a component for you if it does not yet exist.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:auto-create&gt;This markup will only be rendered if that component does not exist.&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In our example we now create 'my-component' upon first rendering of the template.
Because we did not specify an explicit component type, the default type will be used: a HTML &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; will be created.
The processed body of the &lt;em&gt;div&lt;/em&gt; element will be set as the content of our text component.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;As with the component type, because we did not specify an explicit scope, the component will be added to the default scope: usually the asset being rendered.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Specifying component type&lt;/div&gt;
&lt;p&gt;Adding a component type is done with the &lt;em&gt;wcm:type&lt;/em&gt; attribute.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:auto-create wcm:type="rich-text"&gt;This markup will only be rendered if that component does not exist.&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;We still create a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt;, except it will now be of rich-text type.
The value of the &lt;em&gt;wcm:type&lt;/em&gt; attribute must be a known &lt;code&gt;WebCmsComponentType&lt;/code&gt; type key.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
The component type you want to create must have a registered &lt;code&gt;WebCmsComponentAutoCreateStrategy&lt;/code&gt; for auto-creation to be successful.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Specifying creation scope&lt;/div&gt;
&lt;p&gt;The &lt;em&gt;wcm:auto-create&lt;/em&gt; attribute can optionally have a value.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:auto-create="global"&gt;This markup will only be rendered if that component does not exist.&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In our example we look for &lt;em&gt;my-component&lt;/em&gt; in the default scope and all its parents (including global).
If the component is not found, we now auto-create it in the &lt;em&gt;global&lt;/em&gt; scope instead of the default.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock warning"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-warning" title="Warning"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
You can combine the use of &lt;em&gt;wcm:scope&lt;/em&gt; with a scope in &lt;em&gt;wcm:auto-create&lt;/em&gt;.
Be careful though because if you auto-create the component in a scope that is in fact not searched for the component, you will re-create on every request.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_using_placeholders_in_a_markup_component"&gt;&lt;a class="anchor" href="#_using_placeholders_in_a_markup_component"&gt;&lt;/a&gt;Using placeholders in a markup component&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can define placeholder sections in your template and allow other components to include them.
Using placeholders is handy for fixed dynamic content that is not a component in itself, but you would like to provide some flexibility on positioning the content.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:parse-placeholders&gt;
    Template content&#8230;&#8203;
    &lt;div wcm:placeholder="my-placeholder"&gt;Placeholder content&lt;/div&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;If you want your component to access placeholder content from the template, you must attribute your component element with &lt;em&gt;wcm:parse-placeholders&lt;/em&gt;.
When present, the original template markup will always be processed to generate the placeholder content.
There is no limit to the number of placeholders defined in a segment, but be aware that those placeholders are only available within that section (eg. during the rendering of &lt;em&gt;my-component&lt;/em&gt;).&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Any element attributed with &lt;em&gt;wcm:placeholder&lt;/em&gt; defines placeholder content.
The attribute value is the name of the placeholder.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The element on which the attribute is present is also part of the placeholder content.
In the example above the placeholder content would be: &lt;code&gt;&lt;div&gt;Placeholder content&lt;/div&gt;&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Rendering placeholder content in markup components&lt;/div&gt;
&lt;p&gt;The presence of &lt;em&gt;wcm:parse-placeholders&lt;/em&gt; ensures that placeholder content will be processed and made available during rendering.
Rendering the actual placeholder however is always up to the component.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Markup components can render placeholders by using &lt;a href="#components-placeholder-content-markers"&gt;placeholder content markers&lt;/a&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Assume &lt;em&gt;my-component&lt;/em&gt; is a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; with the following content:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-text hljs" data-lang="text"&gt;My placeholder: @@wcm:placeholder(my-placeholder)@@
My other placeholder: @@wcm:placeholder(my-other-placeholder)@@&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Upon rendering the template fragment specified above, the following output would be the result:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div&gt;
    My placeholder: &lt;div&gt;Placeholder content&lt;/div&gt;
    My other placeholder:
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Because there is no placeholder content &lt;em&gt;my-other-placeholder&lt;/em&gt; defined, an empty string is rendered.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Example auto-creation of markup with a placeholder&lt;/div&gt;
&lt;p&gt;When rendering an existing component all markup outside the placeholders is simply ignored.
When auto-creating the component however, that markup is still used to generate the default content of the component.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Assume we auto-create our component:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:parse-placeholders wcm:auto-create&gt;
    Template content&#8230;&#8203;
    &lt;div wcm:placeholder="my-placeholder"&gt;Placeholder content&lt;/div&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;This would result in a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; with the following content:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-text hljs" data-lang="text"&gt;Template content&#8230;&#8203;
@@wcm:placeholder(my-placeholder)@@&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_nesting_components"&gt;&lt;a class="anchor" href="#_nesting_components"&gt;&lt;/a&gt;Nesting components&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Like with placeholders, a markup component can include other components using &lt;a href="#components-component-content-markers"&gt;component content markers&lt;/a&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Assume you have a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; with the following content: &lt;code&gt;My component: @@wcm:component(header,global,false)@@&lt;/code&gt;.&lt;br&gt;
And on the global level the &lt;em&gt;header&lt;/em&gt; component is a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; with content &lt;code&gt;my header&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;When rendering the first component, the output would be &lt;code&gt;My component: my header&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Component content marker parameters&lt;/div&gt;
&lt;p&gt;A component content marker always requires 3 attributes that are equivalents of the Thymeleaf dialect attributes:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;component name (equivalent of &lt;em&gt;wcm:component&lt;/em&gt;)&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;initial scope to look for the component (equivalent of &lt;em&gt;wcm:scope&lt;/em&gt;)&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;true/false if parent scopes should or should not be searched (equivalent of &lt;em&gt;wcm:search-parent-scopes&lt;/em&gt;)&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;If a component is not found, an empty string is added to the output and the marker removed.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Auto-create nested components&lt;/div&gt;
&lt;p&gt;When nesting components in template markup, nested components will always be replaced by a component content marker.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The following markup:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:auto-create&gt;
    My title: &lt;span wcm:component="title"&gt;title&lt;/span&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Would result in a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; with the content &lt;code&gt;My title: &lt;span&gt;@@wcm:component(title,default,true)@@&lt;/span&gt;&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
Because &lt;em&gt;my-component&lt;/em&gt; is not a container, component &lt;em&gt;title&lt;/em&gt; will not get auto-created unless it is in turn attributed with &lt;em&gt;wcm:auto-create&lt;/em&gt;.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Including nested component output&lt;/div&gt;
&lt;p&gt;In some cases you don&#8217;t want to include a content marker for another component, but include the actual component output instead.
You can do so by adding the &lt;em&gt;wcm:parent-create-include&lt;/em&gt; attribute.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Let&#8217;s change our example markup to the following:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-component" wcm:auto-create&gt;
    My title: &lt;span wcm:component="title" wcm:parent-create-include&gt;title&lt;/span&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Assume component &lt;em&gt;title&lt;/em&gt; is a &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; with &lt;code&gt;Some title&lt;/code&gt; as content.
Upon first rendering &lt;em&gt;my-component&lt;/em&gt; would get created with the &lt;em&gt;title&lt;/em&gt; component output included: &lt;code&gt;My title: &lt;span&gt;Some title&lt;/span&gt;&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_auto_create_a_simple_container"&gt;&lt;a class="anchor" href="#_auto_create_a_simple_container"&gt;&lt;/a&gt;Auto-create a simple container&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Apart from simple markup components like &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; you can also auto-create &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt; components.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Let&#8217;s change our example markup to the following:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-container" wcm:type="container" wcm:auto-create&gt;
    &lt;th:block wcm:component="title"&gt;Title&lt;/th:block&gt;
    &lt;th:block wcm:component="body"&gt;Body&lt;/th:block&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Rendering the above example will create &lt;em&gt;my-container&lt;/em&gt; as a &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt;.
The container will have 2 members: &lt;em&gt;title&lt;/em&gt; and &lt;em&gt;body&lt;/em&gt;, both being &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; implementations with their respective processed template markup as content.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
Because &lt;em&gt;title&lt;/em&gt; and &lt;em&gt;body&lt;/em&gt; are component children within a container type, they do not require the &lt;em&gt;wcm:auto-create&lt;/em&gt; attribute themselves.
It is assumed they should be created automatically as members of the container - if the container itself gets auto-created.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_nesting_container_components"&gt;&lt;a class="anchor" href="#_nesting_container_components"&gt;&lt;/a&gt;Nesting container components&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You&#8217;re not limited to using a single level of containers for auto-creation.
Consider the following example:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-container" wcm:type="container" wcm:auto-create&gt;
    &lt;th:block wcm:component="title"&gt;Title&lt;/th:block&gt;
    &lt;th:block wcm:component="body" wcm:type="container"&gt;
        &lt;th:block wcm:component="intro"&gt;Intro&lt;/th:block&gt;
        &lt;th:block wcm:component="main-text"&gt;Main text&lt;/th:block&gt;
    &lt;/th:block&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In this case the following components would be created:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;em&gt;my-container&lt;/em&gt; as &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;title&lt;/em&gt; as &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;body&lt;/em&gt; as &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;intro&lt;/em&gt; as &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;main-text&lt;/em&gt; as &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;No additional &lt;em&gt;wcm:auto-create&lt;/em&gt; attributs are required as all nested components have a container as direct parent.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_using_placeholders_in_containers"&gt;&lt;a class="anchor" href="#_using_placeholders_in_containers"&gt;&lt;/a&gt;Using placeholders in containers&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Much like a regular markup component, a container can also use placeholders that are defined in the template.
Where a markup component uses a &lt;a href="#components-placeholder-content-markers"&gt;placeholder content markers&lt;/a&gt; to render the placeholder content, a &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt; will get a member component of type &lt;code&gt;PlaceholderWebCmsComponentModel&lt;/code&gt; instead.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-container" wcm:type="container" wcm:auto-create&gt;
    &lt;th:block wcm:component="title"&gt;Title&lt;/th:block&gt;
    &lt;div wcm:placeholder="body"&gt;
        &lt;div wcm:component="footer" /&gt;
    &lt;/th:block&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;This would auto-create the following components:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;em&gt;my-container&lt;/em&gt; as &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;title&lt;/em&gt; as &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;body&lt;/em&gt; as &lt;code&gt;PlaceholderWebCmsComponentModel&lt;/code&gt; with &lt;em&gt;body&lt;/em&gt; as the placeholder name&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
In the above example the &lt;em&gt;footer&lt;/em&gt; component reference is outside of the container section as it is inside the placeholder block.
This means that all ties with the container will be severed: the normal scope lookup will apply and the component will not get auto-created unless it also has the &lt;em&gt;wcm:auto-create&lt;/em&gt; attribute.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_linking_to_other_components"&gt;&lt;a class="anchor" href="#_linking_to_other_components"&gt;&lt;/a&gt;Linking to other components&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;A &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt; can hold &lt;code&gt;ProxyWebCmsComponentModel&lt;/code&gt; members that refer to other components that are not container members.
If your template refers to a scoped component inside a container, a proxy will get auto-created as well.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-container" wcm:type="container" wcm:auto-create&gt;
    &lt;th:block wcm:component="title"&gt;Title&lt;/th:block&gt;
    &lt;th:block wcm:component="footer" wcm:scope="global"&gt;Replace by the global footer&lt;/th:block&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In this case member &lt;em&gt;footer&lt;/em&gt; would be a &lt;code&gt;ProxyWebCmsComponentModel&lt;/code&gt; that is linked to the global component with the name &lt;em&gt;footer&lt;/em&gt;.
If the global &lt;em&gt;footer&lt;/em&gt; component is not found however, no member would have been created either, as a proxy only links to an existing component.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can of course still auto-create the global component as well - just as if it were outside a container:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-container" wcm:type="container" wcm:auto-create&gt;
    &lt;th:block wcm:component="title"&gt;Title&lt;/th:block&gt;
    &lt;th:block wcm:component="footer" wcm:scope="global" wcm:auto-create&gt;Replace by the global footer&lt;/th:block&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Now the global &lt;em&gt;footer&lt;/em&gt; component would first get created if it doesn&#8217;t exist yet and then a proxy member would be added to the container.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_containers_with_markup"&gt;&lt;a class="anchor" href="#_containers_with_markup"&gt;&lt;/a&gt;Containers with markup&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Unless a container has markup support active, all template markup outside of &lt;em&gt;wcm:component&lt;/em&gt; blocks will simply be ignored.
If markup is supported however, the markup will contain component content markers much like in the case of markup components.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;However, in the case of container markup, only component content markers will be added that refer to container members.
The container members in turn might be &lt;code&gt;ProxyWebCmsComponentModel&lt;/code&gt; instances or might be &lt;code&gt;PlaceholderWebCmsComponentModel&lt;/code&gt; instances.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;A full example for a markup supporting container:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;div wcm:component="my-container" wcm:type="markup-container" wcm:auto-create&gt;
    Title: &lt;th:block wcm:component="title"&gt;Title&lt;/th:block&gt;
    &lt;div wcm:placeholder="body"&gt;
        &lt;div wcm:component="footer" wcm:scope="global" /&gt;
    &lt;/th:block&gt;
    Footer: &lt;th:block wcm:component="footer" wcm:scope="global"&gt;Replace by the global footer&lt;/th:block&gt;
&lt;/div&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;This would auto-create:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;em&gt;my-container&lt;/em&gt; as &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;title&lt;/em&gt; as &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;body&lt;/em&gt; as &lt;code&gt;PlaceholderWebCmsComponentModel&lt;/code&gt; with &lt;em&gt;body&lt;/em&gt; as the placeholder name&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;member: &lt;em&gt;footer&lt;/em&gt; as &lt;code&gt;ProxyWebCmsComponentModel&lt;/code&gt; linked to the global &lt;em&gt;footer&lt;/em&gt; component (same as is rendered inside the placeholder)&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The markup of &lt;em&gt;my-container&lt;/em&gt; would only link to container members:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-text hljs" data-lang="text"&gt;Title: @@wcm:component(title,container,false)@@
@@wcm:component(body,container,false)@@
Footer: @@wcm:component(footer,container,false)@@&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="components-thymeleaf-rendering"&gt;&lt;a class="anchor" href="#components-thymeleaf-rendering"&gt;&lt;/a&gt;1.1. Custom rendering of component&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Web components in Thymeleaf are rendered using a &lt;code&gt;WebCmsComponentModelRenderer&lt;/code&gt; implementation.
You can easily create your own implementation for custom rendering:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;create your own implementation of &lt;code&gt;WebCmsComponentModelRenderer&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;implement the &lt;code&gt;supports()&lt;/code&gt; method to ensure it is used for the correct types&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;register your implementation as a bean so it can be picked up by the rendering infrastructure&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
If you want to override the default rendering, you must ensure your implementation is registered &lt;strong&gt;before&lt;/strong&gt; the default implementations.
You can do so by ordering your beans (using an &lt;code&gt;@Order&lt;/code&gt; annotation or implementing the &lt;code&gt;Ordered&lt;/code&gt; interface).
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;If you want to use content markers in your content snippets, you can use the &lt;code&gt;WebCmsComponentContentModelWriter&lt;/code&gt; to render the content with Thymeleaf.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect1"&gt;
&lt;h2 id="components-thymeleaf-content-markers"&gt;&lt;a class="anchor" href="#components-thymeleaf-content-markers"&gt;&lt;/a&gt;1. Custom content markers&lt;/h2&gt;
&lt;div class="sectionbody"&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Content containing content markers can easily be written to Thymeleaf output using the &lt;code&gt;WebCmsComponentContentModelWriter&lt;/code&gt;.
If you want to implement your own custom content markers you must provide an implementation of &lt;code&gt;WebCmsComponentContentMarkerRenderer&lt;/code&gt; as a bean.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;</p>
</div>
<div class="paragraph">
<p>&lt;div class="sect1"&gt;
&lt;h2 id="components-defining-component-types"&gt;&lt;a class="anchor" href="#components-defining-component-types"&gt;&lt;/a&gt;Defining component types&lt;/h2&gt;
&lt;div class="sectionbody"&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can easily define your own component types.
Depending on your needs you can &lt;a href="#components-extending-base-types"&gt;extend one of the base types&lt;/a&gt; or create a &lt;a href="#components-custom-component-type"&gt;fully custom implementation&lt;/a&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="sect2"&gt;
&lt;h3 id="components-extending-base-types"&gt;&lt;a class="anchor" href="#components-extending-base-types"&gt;&lt;/a&gt;Extending the base types&lt;/h3&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can add a custom &lt;code&gt;WebCmsComponentType&lt;/code&gt; by importing it via YAML or by adding it via the &lt;code&gt;WebCmsComponentTypeRepository&lt;/code&gt;.
Component types to import should be defined in the section &lt;strong&gt;types - component&lt;/strong&gt; where the key of a component type definition is the`unique &lt;code&gt;WebCmsComponentType&lt;/code&gt; &lt;code&gt;typeKey&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Example YAML component type definition&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-yaml hljs" data-lang="yaml"&gt;types:
  component:
    my-component:
      name: Custom component
      description: Custom HTML component with additional metadata.
      attributes:
        type: markup
        metadata: "my.component.MyMetadata"&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;A &lt;code&gt;WebCmsComponentType&lt;/code&gt; has a collection of attributes.
This is a map of &lt;code&gt;String&lt;/code&gt; key/value pairs.
Adding attributes is a way to provide processing metadata without having to create custom &lt;code&gt;WebCmsComponentType&lt;/code&gt; implementations.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Default attributes&lt;/div&gt;
&lt;p&gt;All base types support 3 default attributes that pack a lot of punch in creating custom component types:&lt;/p&gt;
&lt;/div&gt;
&lt;table class="tableblock frame-all grid-all spread"&gt;
&lt;colgroup&gt;
&lt;col style="width: 20%;"&gt;
&lt;col style="width: 80%;"&gt;
&lt;/colgroup&gt;
&lt;thead&gt;
&lt;tr&gt;
&lt;th class="tableblock halign-left valign-top"&gt;Attribute name&lt;/th&gt;
&lt;th class="tableblock halign-left valign-top"&gt;Description&lt;/th&gt;
&lt;/tr&gt;
&lt;/thead&gt;
&lt;tbody&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;&lt;em&gt;type&lt;/em&gt;&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Contains the base type for the component type.&lt;/p&gt;
&lt;p class="tableblock"&gt; Examples: &lt;em&gt;container&lt;/em&gt;, &lt;em&gt;markup&lt;/em&gt; &#8230;&#8203;&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;&lt;em&gt;template&lt;/em&gt;&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Optional: identifier of the (Thymeleaf) template that should be used for rendering the resulting &lt;code&gt;WebCmsComponentModel&lt;/code&gt; for a component of this type.&lt;/p&gt;
&lt;p class="tableblock"&gt;Example: &lt;em&gt;th/mymodule/mytemplate :: fragment&lt;/em&gt;&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;tr&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;&lt;em&gt;metadata&lt;/em&gt;&lt;/p&gt;&lt;/td&gt;
&lt;td class="tableblock halign-left valign-top"&gt;&lt;p class="tableblock"&gt;Optional: name of the class that should be used for the component metadata.&lt;/p&gt;
&lt;p class="tableblock"&gt;Example: &lt;em&gt;my.module.components.MyComponentMetadata&lt;/em&gt;&lt;/p&gt;&lt;/td&gt;
&lt;/tr&gt;
&lt;/tbody&gt;
&lt;/table&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Custom template&lt;/div&gt;
&lt;p&gt;Any component type can have a &lt;em&gt;template&lt;/em&gt; attribute set that identifies the (Thymeleaf) template that should be used for rendering the component instances.
If a template is specified it will always be used for rendering and any other &lt;code&gt;WebCmsComponentModelRenderer&lt;/code&gt; for that component type will be skipped.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can customize rendering the default &lt;code&gt;WebCmsComponentModel&lt;/code&gt; implementations just by specifying a custom template.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
Property &lt;code&gt;renderTemplate&lt;/code&gt; on &lt;code&gt;WebCmsComponentModel&lt;/code&gt; allows you to set a custom template per-component instead of per component type.
If a &lt;code&gt;renderTemplate&lt;/code&gt; is set on the component itself, it will take precedence over the &lt;code&gt;WebCmsComponentType&lt;/code&gt; template attribute.
Setting &lt;code&gt;renderTemplate&lt;/code&gt; is not enabled by default in the administration UI.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Metadata&lt;/div&gt;
&lt;p&gt;Any &lt;code&gt;WebCmsComponentModel&lt;/code&gt; has a &lt;code&gt;metadata&lt;/code&gt; property that refers to an object that has all custom metadata for a component.
The type of metadata object is configured by setting the class name as the &lt;em&gt;metadata&lt;/em&gt; attribute on the &lt;code&gt;WebCmsComponentType&lt;/code&gt;.
The metadata instance will always be created as a prototype using the &lt;code&gt;BeanFactory&lt;/code&gt; and as such supports autowiring other beans from the &lt;code&gt;ApplicationContext&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;A metadata class must:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;be serializable to and from JSON using Jackson &lt;code&gt;ObjectMapper&lt;/code&gt;&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;Jackson annotations on the metadata class are supported if necessary to define (de-)serializer rules&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;special care should be taken not to serializer any beans that might have been wired&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;implement a valid &lt;code&gt;equals()&lt;/code&gt; method in order to have change detection and better import/export performance&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;be easily cloneable:&lt;/p&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;have either a no-arguments public constructor and provide setters/getters for all relevant properties&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;OR implement &lt;code&gt;EntityWithDto&lt;/code&gt; or &lt;code&gt;Cloneable&lt;/code&gt;&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;If the administration UI is active, a form for metadata properties will automatically be generated using the EntityModule.
This does require you to register the metadata type as an entity in the &lt;code&gt;EntityRegistry&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
Default (de-)serialization for &lt;code&gt;WebCmsObject&lt;/code&gt; implementations is provided by the {module-name} by dispatching to the &lt;code&gt;WebCmsDataConversionService&lt;/code&gt;.
This means metadata properties can often refer to other web cms types without requiring custom configuration.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;See &lt;a href="#components-custom-component-type"&gt;creating a custom component type&lt;/a&gt; if you want to have a custom metadata class, change serialization or admin UI rendering.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;div class="title"&gt;Providing a component template&lt;/div&gt;
&lt;p&gt;When defining a custom component type, you can link a &lt;strong&gt;componentTemplate&lt;/strong&gt; component to the &lt;code&gt;WebCmsComponentType&lt;/code&gt; definition.
If present, that component will be used as a template for a new component of your custom component type.
Class properties as well as any child components will be copied to a new component your type.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;When your component type and &lt;strong&gt;componentTemplate&lt;/strong&gt; are a container type, any &lt;code&gt;TextWebCmsComponentModel&lt;/code&gt; member of the template component can have special markers in its content.
These markers will be replaced when copying the template to a newly created component:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@container.name@@&lt;/code&gt; will be replaced with the &lt;code&gt;name&lt;/code&gt; of the newly created container&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;code&gt;@@container.title@@&lt;/code&gt; will be replaced with the &lt;code&gt;title&lt;/code&gt; of the newly created container&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
The &lt;strong&gt;componentTemplate&lt;/strong&gt; should have the same class implementation as the component type being defined.
For example when creating a &lt;strong&gt;fixed-container&lt;/strong&gt;, the &lt;strong&gt;componentTemplate&lt;/strong&gt; must implement &lt;code&gt;ContainerWebCmsComponentModel&lt;/code&gt;.
Providing custom metadata is not supported on the template component, the metadata will be reset when creating a new component.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_example_custom_container_component"&gt;&lt;a class="anchor" href="#_example_custom_container_component"&gt;&lt;/a&gt;Example: custom container component&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;This is an example of a custom component that is a fixed container with 2 member components.
A custom metadata class allows configuring the position of the members (left or right), the custom template uses the metadata to determine what the output should be.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Component definition&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-yaml hljs" data-lang="yaml"&gt;types:
  component:
    left-right:
      name: Left or Right
      description: Renders 2 components in a specific layout.
      attributes:
        type: fixed-container
        template: "th/mymodule/components :: left-right"
        metadata: "mymodule.LeftRightMetadata"
      wcm:components:
        componentTemplate:
          componentType: container
          wcm:components:
            one:
              componentType: rich-text
              title: One
              sortIndex: 1
              body: "@@container.name@@"
            two:
              componentType: rich-text
              title: Two
              sortIndex: 2
              body: "@@container.title@@"&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;When creating a new &lt;em&gt;left-right&lt;/em&gt; component the members of &lt;em&gt;componentTemplate&lt;/em&gt; will be cloned into the new container.
The text components &lt;em&gt;one&lt;/em&gt; and &lt;em&gt;two&lt;/em&gt; will get their default content set with respectively the name and title of the new &lt;em&gt;left-right&lt;/em&gt; component.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The &lt;em&gt;sortIndex&lt;/em&gt; will determine the order of the components in the administration UI.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Metadata class&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-java hljs" data-lang="java"&gt;@Data   // Use Lombok @Data to generate getters, setters and equals() method
public class LeftRightMetadata
{
    enum Layout
    {
        LEFT_TO_RIGHT,
        RIGHT_TO_LEFT
    }</p>
</div>
<div class="literalblock">
<div class="content">
<pre>    /**
     * Determines the order of rendering one and two.
     */
    @NotNull
    private Layout layout = Layout.LEFT_TO_RIGHT;
}</pre>
</div>
</div>
<div class="paragraph">
<p>@Configuration
@ConditionalOnAdminUi
class LeftRightConfiguration implements EntityConfigurer {
    @Override
    public void configure( EntitiesConfigurationBuilder entities ) {
        entities.create().entityType( LeftRightMetadata.class, true );
    }
}&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The metadata only has a single property &lt;code&gt;layout&lt;/code&gt;.
When the administration UI is active (presence of EntityModule and AdminWebModule) the layout value can be selected in the user interface.
The default administration UI uses the the EntityModule to build the metadata form, so we register the &lt;code&gt;LeftRightMetadata&lt;/code&gt; as an entity.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="admonitionblock note"&gt;
&lt;table&gt;
&lt;tr&gt;
&lt;td class="icon"&gt;
&lt;i class="fa icon-note" title="Note"&gt;&lt;/i&gt;
&lt;/td&gt;
&lt;td class="content"&gt;
If you do not want to use the EntityModule to generate the metadata form, you can provide a custom &lt;code&gt;WebCmsComponentModelMetadataAdminRenderer&lt;/code&gt;.
See &lt;a href="<mark>components-custom-component-type"&gt;create a custom component type&lt;/a&gt; for more information.
&lt;/td&gt;
&lt;/tr&gt;
&lt;/table&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Thymeleaf template: th/mymodule/components&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-html hljs" data-lang="html"&gt;&lt;th:block th:fragment="left-right(component)"
          th:with="metadata=${component.metadata}"&gt;
    &lt;section th:if="${metadata.layout.name() eq 'LEFT_TO_RIGHT'}"&gt;
        &lt;div class="left"&gt;&lt;across:view element="${component.getMember('one')}" /&gt;&lt;/div&gt;
        &lt;div class="right"&gt;&lt;across:view element="${component.getMember('two')}" /&gt;&lt;/div&gt;
    &lt;/section&gt;
    &lt;section th:if="${metadata.layout.name() eq 'RIGHT_TO_LEFT'}"&gt;
        &lt;div class="left"&gt;&lt;across:view element="${component.getMember('two')}" /&gt;&lt;/div&gt;
        &lt;div class="right"&gt;&lt;across:view element="${component.getMember('one')}" /&gt;&lt;/div&gt;
    &lt;/section&gt;
&lt;/th:block&gt;&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;The Thymeleaf template inspects the metadata &lt;code&gt;layout&lt;/code&gt; property and renders members &lt;em&gt;one&lt;/em&gt; and &lt;em&gt;two&lt;/em&gt; in a fixed location.
As a &lt;code&gt;WebCmsComponentModel&lt;/code&gt; is a &lt;code&gt;ViewElement&lt;/code&gt; using an &lt;em&gt;across:view&lt;/em&gt; node takes care of rendering the member components.&lt;/p&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="sect2"&gt;
&lt;h3 id="_filtering_selectable_webcmscomponenttype_options"&gt;&lt;a class="anchor" href="#_filtering_selectable_webcmscomponenttype_options"&gt;&lt;/a&gt;Filtering selectable WebCmsComponentType options&lt;/h3&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;To filter the provided dropdown options for a WebCmsComponent, the user can perform two actions:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;options that should never be present by default&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;selected set of options that should be present.&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;These actions are provided through the &lt;code&gt;DefaultAllowedComponentTypeFetcher&lt;/code&gt;. Should you like to provide your own logic to filter the selectable options, for one or more types, you will have to provide your own implementations of the &lt;code&gt;WebCmsAllowedComponentTypeFetcher&lt;/code&gt; interface.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="sect3"&gt;
&lt;h4 id="_non_selectable_webcmscomponenttypes"&gt;&lt;a class="anchor" href="#_non_selectable_webcmscomponenttypes"&gt;&lt;/a&gt;Non-selectable WebCmsComponentTypes&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;To prevent a &lt;code&gt;WebCmsComponentType&lt;/code&gt; from showing up in the list unless explicitly specified, the user has to provide the component type with the &lt;code&gt;componentRestricted&lt;/code&gt; attribute, with its value set to true. This ensures that unless the WebCmsComponentType is specified as an option for the &lt;code&gt;WebCmsObject&lt;/code&gt; through a &lt;code&gt;WebCmsTypeSpecifierLink&lt;/code&gt;, it will never be shown as an option.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Examples of non-selectable WebCmsComponentTypes are the default placeholder, proxy and fixed-container component types.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Example&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-yaml hljs" data-lang="yaml"&gt;types:
  component:
    fixed-container:
      objectId: "wcm:type:component:fixed-container"
      name: Fixed container
      description: Container of a fixed set of components.
      attributes:
        type: fixed-container
        componentRestricted: true&lt;/code&gt;&lt;/pre&gt;
&lt;/div&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;A &lt;em&gt;fixed-container&lt;/em&gt; should never be available as a selectable option unless specified (through a &lt;code&gt;WebCmsTypeSpecifierLink&lt;/code&gt;)&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;/div&gt;

&lt;h4 id="_selectable_webcmscomponenttype_options"&gt;&lt;a class="anchor" href="#_selectable_webcmscomponenttype_options"&gt;&lt;/a&gt;Selectable WebCmsComponentType options&lt;/h4&gt;

&lt;p&gt;All component related interfaces use the same processing approach:&lt;/p&gt;
&lt;/div&gt;

&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;all beans of that type are detected and ordered&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;a &lt;code&gt;supports()&lt;/code&gt; method is used to check if the bean applies for a certain component&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;the first bean that applies will be used&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;As the implementations are always queried in order, customizing an implementation is a matter of:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;providing your implementation as a bean&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;implementing &lt;code&gt;supports()&lt;/code&gt; to match for all component types you want&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;ensure your bean is ordered (use &lt;code&gt;@Order&lt;/code&gt; or implement &lt;code&gt;Ordered&lt;/code&gt;) before any other that might also apply for that component type&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;

&lt;h4 id="components-component-profiles"&gt;&lt;a class="anchor" href="#components-component-profiles"&gt;&lt;/a&gt;Configuring rich-text and markup components&lt;/h4&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;WebCmsModule supports rich text components and HTML components out of the box, using TinyMCE and CodeMirror respectively.  Rich-text components are identified with the base type &lt;code&gt;rich-text&lt;/code&gt;, markup with &lt;code&gt;markup&lt;/code&gt;.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;YAML definition of default Rich text and HTML component types&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-yaml hljs" data-lang="yaml"&gt;rich-text:
  objectId: "wcm:type:component:rich-text"
  name: Rich text
  description: Rich text component - supporting HTML markup and default assets if enabled by other modules.
  attributes:
    type: rich-text
    profile: rich-text

  name: HTML
  description: A snippet of HTML.
  attributes:

&lt;/div&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;In addition to the base type, a component type can define a &lt;strong&gt;profile&lt;/strong&gt; attribute value, denoting a custom profile that should be used to initialize the editor for that component type.
The default rich-text component type uses a profile called &lt;code&gt;rich-text&lt;/code&gt; as well.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;You can customize the profile configuration using Javascript.  This is done by registering profiles on the &lt;code&gt;WebCmsComponentProfileRegistry&lt;/code&gt;.  A profile registration requires the following three attributes:&lt;/p&gt;
&lt;/div&gt;
&lt;div class="ulist"&gt;
&lt;ul&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;componentType&lt;/strong&gt;: this is the base type of the component (eg. &lt;code&gt;rich-text&lt;/code&gt; or &lt;code&gt;markup&lt;/code&gt;)&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;profileName&lt;/strong&gt;: a unique profile name&lt;/p&gt;
&lt;/li&gt;
&lt;li&gt;
&lt;p&gt;&lt;strong&gt;profile&lt;/strong&gt;: JSON object that holds the actual profile data (for example the TinyMCE configuration object)&lt;/p&gt;
&lt;/li&gt;
&lt;/ul&gt;
&lt;/div&gt;
&lt;div class="paragraph"&gt;
&lt;p&gt;Profiles can be inherited.  If your profile data has a &lt;strong&gt;_parentProfile&lt;/strong&gt; attribute, the value is expected to be the name of another profile for the same component type.  The final profile will be the data of the parent profile merged together with the requested profile.  Values from the parent will be replaced by values from the requested profile. Every profile can have a (optional) single parent.&lt;/p&gt;
&lt;/div&gt;
&lt;div class="listingblock"&gt;
&lt;div class="title"&gt;Example registration of custom rich-text type&lt;/div&gt;
&lt;div class="content"&gt;
&lt;pre class="highlightjs highlight"&gt;&lt;code class="language-javascript hljs" data-lang="javascript"&gt;(function ( $ ) {
  /<strong>*
   * Limited rich-text profile: inherit from the _base profile
   * and only show bold/italic buttons.
   */
  WebCmsComponentProfileRegistry.registerProfileForComponentType(
    'rich-text', /</strong> component type <strong>/
    'limited', /</strong> profile name */
    {
      _parentProfile: '_base',
      toolbar1: 'bold italic'
    }

})(jQuery);&lt;/code&gt;&lt;/pre&gt;
